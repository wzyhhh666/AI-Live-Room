# C++ 直播弹幕+房间系统 — 性能优化方案

## 1. 目标

本文件定义性能目标、压测方式和分阶段优化路线。代码开发前可以先看目标值，避免优化方向跑偏。

---

## 2. 性能指标定义

| 指标 | 含义 |
|------|------|
| QPS | 每秒处理请求数或弹幕数 |
| 延迟 | 单条消息端到端处理耗时 |
| 连接数 | 同时在线 TCP 连接数量 |
| 内存占用 | 单机总内存和单连接内存 |
| CPU 占用 | 网络层与业务层的 CPU 压力 |
| 丢包率 | 高并发下失败消息比例 |

### 建议目标

- 单机长连接：5 万+
- 单条弹幕端到端延迟：< 50ms
- 单机弹幕吞吐：10 万条/秒
- 敏感词过滤耗时：< 1ms/条

---

## 3. 压测方案与工具

### 3.1 压测方式

- 自研 TCP 压测客户端，模拟大量用户在线和发弹幕。
- 分别压连接、压广播、压过滤、压存储。
- 单项压测和全链路压测都要做。

### 3.2 常用工具

- `perf`：CPU 火焰图与热点分析。
- `valgrind` / `heaptrack`：内存问题分析。
- `gprof`：传统性能统计。
- `flamegraph`：调用栈火焰图。
- 自研 bench 工具：网络和业务压测主工具。

### 3.3 压测记录

每次压测都必须记录：

- 版本号
- 配置参数
- 并发量
- 消息量
- 延迟分布
- 错误率

## 4. 分阶段优化路线

### V1.0 基础可用

- Reactor + 线程池打通。
- 房间创建、进入、发弹幕闭环跑通。
- 基础数据库持久化完成。

### V2.0 稳定性优化

- 连接管理优化。
- 定时器优化。
- 日志和监控补齐。
- 错误处理和补偿机制完善。

### V3.0 吞吐优化

- 广播路径减少拷贝。
- 任务队列优化。
- 热点容器优化。
- 批量写入数据库。

### V4.0 资源优化

- 内存池 / 对象池。
- 缓冲区复用。
- 降低锁粒度。
- 减少频繁分配释放。

### V5.0 架构优化

- 分片、拆房间或多节点扩展。
- 需要时再引入消息中间件。
- 为分布式做接口预留。

---

## 5. 优化重点

- 优先优化最热路径：接收、过滤、广播。
- 先定位瓶颈，再决定改结构还是改算法。
- 任何优化都必须有压测前后对比数据。

---

## 6. 不要做的事

- 不要没压测就先改成复杂无锁结构。
- 不要为了“性能看起来高”提前做分布式。
- 不要在没有数据证明的情况下频繁重构。
