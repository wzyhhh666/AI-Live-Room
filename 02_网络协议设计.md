# C++ 直播弹幕+房间系统 — 网络协议设计

## 0. 设计目标

本文件定义所有网络交互的唯一协议标准。AI 生成协议代码时，必须与这里完全一致，不能自创字段名、消息类型或包格式。

---

## 1. 传输层

- 传输协议固定为 **TCP 长连接**。
- 客户端与服务端在登录后保持连接。
- 所有业务消息都通过同一条 TCP 连接收发。
- 心跳消息用于维持连接存活，超时则主动断开。

---

## 2. 消息格式

采用自定义二进制协议：**Length + Header + Body**。

### 2.1 格式定义

| 字段 | 说明 |
|------|------|
| Length | 后续 Header + Body 的总字节长度 |
| Header | 固定长度消息头 |
| Body | 业务消息体，使用 Protobuf 编码 |

### 2.2 包结构示例

```text
+---------+----------------------+------------------+
| Length  | Header               | Body             |
+---------+----------------------+------------------+
| 4 bytes | fixed 48 bytes      | variable         |
+---------+----------------------+------------------+
```

### 2.3 编码约定

- Length 使用 `uint32_t`。
- 所有多字节字段统一使用网络字节序（big-endian）。
- Header 固定 48 字节，方便快速解析。单次 read 精确读取 4 + 48 = 52 字节即可完成 Length + Header 的完整解析。
- Body 使用 Protobuf，保证扩展性和兼容性。

---

## 3. 消息头定义

建议固定如下字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| magic | uint32_t | 魔数，用于快速识别非法包 |
| version | uint16_t | 协议版本 |
| headerLen | uint16_t | 头部长度 |
| msgType | uint16_t | 消息类型枚举 |
| flags | uint16_t | 标记位，如压缩、加密 |
| seq | uint64_t | 序列号，用于请求响应匹配 |
| roomId | uint64_t | 房间号 |
| userId | uint64_t | 用户号 |
| timestamp | uint64_t | 时间戳，毫秒级 |
| bodyLen | uint32_t | Body 长度 |

### 3.1 设计说明

- `magic` 用于防止脏数据误解析。
- `seq` 用于请求追踪和日志关联。
- `roomId` 用于房间广播路由。
- `userId` 用于鉴权、限流和审计。
- `timestamp` 用于日志对齐、心跳判断和超时判断。
- `bodyLen` 用于拆包与完整性校验。

### 3.2 Header 与 Body 字段关系

`roomId`、`userId` 在 Header 和 Protobuf Body 中可能同时存在，设计意图如下：

- Header 中的 `roomId`、`userId` 用于网络层快速路由和鉴权，不依赖 Body 解析。
- Body 中的 `room_id`、`user_id` 用于业务逻辑的完整语义表达和持久化。

### 3.3 冲突处理规则

- 以 Header 中的 `roomId`、`userId` 为权威值。
- Body 中的 `room_id`、`user_id` 仅作业务参考，不用于路由和鉴权。
- 服务端发送消息时，必须保证 Header 和 Body 中的 `roomId`、`userId` 一致。
- 如果收到的消息 Header 与 Body 不一致，以 Header 为准，并记录 WARN 日志。
- 协议层解析 Header 后，应将 `roomId`、`userId` 注入到业务上下文，业务层直接使用上下文值。

## 4. 消息类型枚举

| 类型 | 说明 |
|------|------|
| `MSG_LOGIN` | 用户登录认证 |
| `MSG_LOGIN_RESP` | 登录响应（含鉴权结果） |
| `MSG_LOGOUT` | 用户主动登出 |
| `MSG_KICK` | 被踢下线（管理员/主播操作） |
| `MSG_DANMAKU` | 发送弹幕 |
| `MSG_GIFT` | 发送礼物 |
| `MSG_JOIN_ROOM` | 进入房间 |
| `MSG_LEAVE_ROOM` | 离开房间 |
| `MSG_HEARTBEAT` | 心跳 |
| `MSG_ROOM_CREATE` | 创建房间 |
| `MSG_ROOM_CLOSE` | 关闭房间 |
| `MSG_ROOM_STATE_SYNC` | 房间状态同步 |
| `MSG_ERROR` | 错误返回 |
| `MSG_ACK` | 通用确认 |

### 4.1 枚举建议

- 使用 `enum class MessageType : uint16_t`。
- 业务消息和系统消息分开编号，避免后续冲突。
- 新增类型时必须先更新协议文档，再改代码。

### 4.2 编号建议

- 系统消息段建议预留给登录、心跳、错误和确认类消息。
- 房间消息段建议预留给进房、退房、房间状态同步类消息。
- 业务消息段建议预留给弹幕、礼物等高频消息。

## 5. Protobuf 消息体设计

所有业务 Body 均使用 `.proto` 定义。

### 5.1 示例 proto

```proto
syntax = "proto3";

package chatroom.protocol;

message DanmakuBody {
  uint64 room_id = 1;
  uint64 user_id = 2;
  string user_name = 3;
  string content = 4;
  uint64 sent_at = 5;
}

message GiftBody {
  uint64 room_id = 1;
  uint64 user_id = 2;
  string gift_name = 3;
  uint32 gift_count = 4;
  uint64 sent_at = 5;
}

message RoomStateBody {
  uint64 room_id = 1;
  uint64 host_id = 2;
  uint32 state = 3;
  string title = 4;
}

message HeartbeatBody {
  uint64 room_id = 1;
  uint64 user_id = 2;
  uint64 client_ts = 3;
}
```

### 5.2 Body 约束

- Body 字段尽量保持语义稳定。
- 新字段只能追加，不能随意修改已有字段编号。
- 删除字段时保留编号空洞，防止协议不兼容。

### 5.3 Proto 文件组织规范

- `proto/common.proto`：公共类型定义，包括通用消息头、错误结构等。
- `proto/auth.proto`：登录和鉴权相关消息体。
- `proto/room.proto`：房间相关消息体。
- `proto/danmaku.proto`：弹幕相关消息体。
- `proto/gift.proto`：礼物相关消息体。

包结构统一为：

```proto
syntax = "proto3";

package chatroom.protocol;
```

命名规则：

- 消息名使用 PascalCase，并以 `Body` 结尾，例如 `DanmakuBody`、`LoginRespBody`。
- 字段名使用 snake_case。
- 字段编号从 1 开始，优先把高频字段放在前面。
- `.proto` 文件引入顺序建议为 `common.proto`、`auth.proto`、`room.proto`、`danmaku.proto`、`gift.proto`。

### 5.4 登录鉴权消息示例

- 密码哈希算法统一使用 SHA-256，客户端发送 `user_name + SHA-256(password)`。

```proto
message LoginBody {
  string user_name = 1;
  string password_hash = 2;
  string client_version = 3;
}

message LoginRespBody {
  uint64 user_id = 1;
  bool success = 2;
  string token = 3;
  string message = 4;
  uint32 role = 5;
}

message LogoutBody {
  uint64 user_id = 1;
  string token = 2;
}

message KickBody {
  uint64 operator_id = 1;
  uint64 target_id = 2;
  uint64 room_id = 3;
  string reason = 4;
}
```

## 6. 粘包与拆包处理策略

### 6.1 处理流程

1. 读取 Length。
2. 判断缓冲区是否达到完整包长度。
3. 读取 Header。
4. 校验 magic、version、bodyLen。
5. 读取 Body 并反序列化 Protobuf。
6. 投递给业务层。

### 6.2 策略要求

- 允许一次读取多个完整包。
- 允许半包缓存后下次继续拼接。
- 非法长度直接丢弃连接。
- 包体长度与实际字节数不一致时直接判为协议错误。

### 6.3 缓冲区建议

- 每个连接维护独立读缓冲。
- 缓冲区必须支持动态扩容，但要设置上限。
- 单连接缓冲超过上限时，视为异常连接并断开。

## 7. 心跳机制

### 7.1 心跳规则

- 客户端每 **10 秒** 发送一次心跳。
- 服务端在 **30 秒** 内未收到心跳，则认为连接超时。
- 连续丢失心跳超过阈值后主动断开连接。

### 7.2 心跳处理

- 心跳包不进入业务线程池，只做在线时间刷新。
- 心跳包必须轻量，Body 只保留必要字段。
- 服务端也可以在空闲时主动下发 ping，用于双向保活。

### 7.3 超时策略

- 优先移除长时间无响应连接。
- 对异常抖动网络允许一次宽限，不要过度踢线。
- 心跳判断必须与定时器模块联动。

---

## 8. 连接与消息安全

- 必须先校验包头，再解析包体。
- 必须校验房间号和用户权限。
- 必须对消息大小做上限限制，防止恶意大包。
- 非法消息类型直接返回错误并记录日志。

---

## 9. 协议演进原则

- 协议版本号必须保留。
- 增量兼容优先，不允许轻易破坏旧客户端。
- 所有协议变更必须同步更新本文件和 `.proto` 文件。

---

## 10. 登录鉴权流程

### 10.1 连接生命周期

- 登录请求统一使用 `user_name + SHA-256(password)` 作为认证输入，其中 `password_hash` 字段承载 SHA-256 后的十六进制字符串。

```text
客户端                         服务端
  │                              │
  │──── TCP Connect ────────────→│  建立 TCP 连接
  │                              │  连接状态 = UNAUTHED
  │                              │
  │──── MSG_LOGIN ─────────────→│  校验用户名/密码
  │                              │  生成 token
  │                              │  绑定 fd → userId
  │                              │  连接状态 = AUTHED
  │←─── MSG_LOGIN_RESP ─────────│  返回 token + 用户信息
  │                              │
  │──── MSG_JOIN_ROOM ────────→│  仅 AUTHED 连接允许
  │──── MSG_DANMAKU ──────────→│  仅 AUTHED 连接允许
  │                              │
  │──── MSG_LOGOUT / 断开 ────→│  解绑 fd → userId
  │                              │  清理资源
```

### 10.2 连接状态定义

```cpp
enum class ConnState : uint8_t {
    UNAUTHED = 0,
    AUTHED = 1,
    KICKED = 2,
};
```

### 10.3 fd 与用户绑定机制

- 每个 Connection 对象内部维护 `userId`，默认 0 表示未登录。
- 登录成功后写入 `userId`、`token`、角色信息。
- 服务端维护全局映射 `std::unordered_map<int, uint64_t> g_fdToUser` 和 `std::unordered_map<uint64_t, int> g_userToFd`。
- 所有业务消息在处理前都必须检查 `Connection.userId != 0`。

### 10.4 Token 设计

- 登录成功后服务端生成唯一 token，建议使用 UUID 或随机 32 字节 hex。
- token 存入 Redis：`session:{token}` → `{userId, userName, role, expireAt}`。
- token 有效期建议 24 小时，心跳时不续期，重新登录刷新。
- 可选地在 Header 的 `flags` 字段中预留 1 bit 标识“携带 token”。

### 10.5 未鉴权连接处理规则

- 未登录连接只能发送 `MSG_LOGIN`、`MSG_HEARTBEAT`。
- 未登录连接发送其他消息类型时，服务端返回 `MSG_ERROR`。
- 未登录连接超过 30 秒仍未完成登录时，主动断开。
