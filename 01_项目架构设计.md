# C++ 直播弹幕+房间系统 — 项目架构设计

## 0. 架构目标

本文件只负责回答三个问题：系统怎么分层、线程怎么跑、模块怎么依赖。代码实现必须严格按照这里的边界来写。

---

## 1. 整体架构

系统采用 **Reactor + 线程池 + 消息队列** 的经典高并发架构。

```
┌──────────────────────────────────────────────────────┐
│                    客户端 (TCP 长连接)                │
└───────────────┬──────────────────────────────────────┘
                │
                ▼
┌──────────────────────────────────────────────────────┐
│               网络层 / Reactor 主线程                │
│        epoll 监听连接、收包、心跳、广播分发           │
└───────────────┬──────────────────────────────────────┘
                │ 解析后消息
                ▼
┌──────────────────────────────────────────────────────┐
│                 消息队列 / 任务分发层                 │
│     连接事件、弹幕事件、房间事件、定时器事件         │
└───────────────┬──────────────────────────────────────┘
                │ 任务对象
                ▼
┌──────────────────────────────────────────────────────┐
│                 工作线程池 / 业务处理层               │
│   弹幕处理、房间状态更新、敏感词过滤、存储写入       │
└───────────────┬──────────────────────────────────────┘
                │
                ▼
┌──────────────────────────────────────────────────────┐
│                     数据层 / 缓存层                   │
│                MySQL + Redis + 本地索引               │
└──────────────────────────────────────────────────────┘
```

---

## 2. 模块划分

### 2.1 网络层

职责：
- 维护 TCP 长连接
- 维护连接状态，拦截未登录连接的非法请求
- 使用 epoll 处理连接事件、可读事件、可写事件
- 完成包头读取、包体读取和基础校验
- 把完整消息投递到消息队列

### 2.2 协议层

职责：
- 定义消息格式、消息头、消息类型枚举
- 提供序列化与反序列化接口
- 处理粘包、拆包、心跳、协议版本兼容

### 2.3 业务层

职责：
- 弹幕广播
- 房间状态管理
- 用户上下线处理
- 敏感词过滤
- 限流、鉴权、消息落库

### 2.4 数据层

职责：
- MySQL 负责持久化数据
- Redis 负责热数据、排行榜、在线状态、缓存
- 本地内存结构负责连接态和临时路由

---

## 3. 线程模型

### 3.1 主线程

主线程只做三件事：

1. 初始化网络资源和 epoll。
2. 监听连接建立、关闭和读写就绪事件。
3. 把完整消息投递给工作线程池，不在主线程里做重业务。

### 3.2 工作线程池

工作线程只做业务处理：

- 弹幕校验与过滤
- 房间状态更新
- 数据持久化
- 广播目标计算
- 发送结果封装

### 3.3 定时线程

定时线程负责：

- 心跳超时检测
- 空闲连接清理
- 房间定时任务
- 排行榜刷新

---

## 4. 依赖关系

### 4.1 单向依赖原则

依赖方向必须始终保持：

`网络层 → 协议层 → 业务层 → 数据层`

### 4.2 禁止事项

- 禁止数据层反向依赖业务层。
- 禁止协议层直接包含业务逻辑。
- 禁止网络层直接访问 MySQL 或 Redis。
- 禁止模块间互相 include 对方实现文件。

### 4.3 依赖方式

- 头文件只暴露接口，不暴露内部实现细节。
- 模块间交互通过抽象接口、回调或 Result 对象完成。
- 复杂耦合通过事件总线或观察者模式解耦。

### 5.1 目录职责

- `include/`：对外暴露的头文件。
- `src/`：实现文件。
- `proto/`：Protobuf 定义文件。
- `config/`：运行配置。
- `tests/`：测试代码。
- `tools/`：压测、运维、辅助工具。

## 6. 第三方库与版本锁定

建议通过 CMake + vcpkg 或 Conan 锁定版本，避免同一项目不同机器编译结果不一致。

| 组件 | 推荐版本 | 用途 |
|------|----------|------|
| spdlog | 1.14+ | 日志 |
| fmt | 11+ | 格式化输出 |
| protobuf | 27+ | 协议序列化 |
| gtest | 1.14+ | 单元测试 |
| yaml-cpp | 0.8+ | 配置解析 |
| nlohmann/json | 3.11+ | JSON 配置和调试 |
| hiredis | 1.2+ | Redis 客户端 |
| mysqlclient / connector | 8.0+ | MySQL 访问 |

### 6.1 锁定原则

- 所有版本写进 CMake 或依赖清单。
- 任何升级必须先验证兼容性，再改文档和构建脚本。
- 不允许同一项目里多人各自手动换版本。

---

## 7. 启动顺序

1. 加载配置。
2. 初始化日志。
3. 初始化数据库和 Redis。
4. 初始化协议和线程池。
5. 初始化 epoll 和网络监听。
6. 启动工作线程和定时线程。
7. 开始接收客户端连接。

---

## 8. 架构约束

- 项目核心闭环必须先跑通“连接 → 发弹幕 → 广播 → 落库 → 查询房间”五个动作。
- 所有模块设计都必须服务于这条主链路。
- 不允许为了“架构漂亮”而提前引入分布式、消息中间件或复杂微服务。
