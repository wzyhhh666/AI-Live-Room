## 1. 客户端开发规范

### 1.1 设计目标

客户端代码的约束优先级与服务端一致：先规范、后实现；先协议、后 UI；先状态、后交互。客户端不直接访问 MySQL 或 Redis，也不绕过协议层拼装业务消息。

### 1.2 推荐标准

- 语言标准固定为 **C++17**。
- 命名风格与服务端保持一致，统一使用 PascalCase、camelCase 和 `m_` 成员前缀。
- 运行期统一使用 `Result<T>` 和错误码，不在消息循环中依赖异常控制流程。
- 日志格式与服务端保持可联调性，至少包含时间、级别、模块、线程 ID、会话状态。

### 1.3 客户端约束

- 客户端只维护本地会话态和展示态，不持有服务端权威数据。
- 网络失败、鉴权失败、房间失效、消息超时都必须有明确可恢复路径。
- 所有界面交互必须落到明确的模块接口，不允许 UI 直接调用底层 socket。

## 2. 客户端架构设计

### 2.1 分层结构

客户端建议采用以下分层：

- 交互层：输入框、列表、状态提示、弹幕展示、房间信息展示。
- 应用层：登录态管理、页面路由、会话状态、重连控制。
- 协议层：消息封包、解包、序列化、反序列化、错误处理。
- 网络层：TCP 连接、收发缓冲、心跳、断线重连。
- 本地缓存层：最近房间、最近消息、未发送队列、展示快照。

### 2.2 线程模型

- 主线程负责 UI 事件和状态刷新。
- 网络线程负责 socket 收发和协议解析。
- 定时线程负责心跳、重连和缓存刷新。
- 业务任务队列负责异步更新展示数据，避免阻塞主线程。

### 2.3 模块边界

- UI 层只订阅状态，不直接操作网络细节。
- 协议层只关心消息结构，不关心页面逻辑。
- 网络层只处理连接和收发，不处理具体业务判断。
- 缓存层只保存可快速恢复的本地状态，不作为权威数据源。

## 3. 客户端网络与协议对接

### 3.1 协议一致性要求

客户端必须与服务端共享同一套协议约定：

- 传输层固定为 TCP 长连接。
- 包格式固定为 `Length + Header + Body`。
- Header 固定 48 字节。
- Body 使用 `proto3` 和 `chatroom.protocol` 包名。
- 消息类型与服务端保持完全一致，不允许客户端私自扩展。

### 3.2 登录与鉴权

- 客户端登录请求使用 `user_name + SHA-256(password)`。
- `password_hash` 字段承载 SHA-256 后的十六进制字符串。
- 登录成功后保存 `userId`、`token`、角色信息和连接状态。
- 客户端后续业务请求必须携带有效会话态，不允许跳过登录直接发房间或弹幕消息。

### 3.3 收发规则

- 接收侧必须先解析 Header，再反序列化 Body。
- 发送侧必须先构造消息类型，再填充 Header 和 Body。
- 未登录状态只能发送登录、心跳相关消息。
- 断线重连后，客户端应优先恢复登录态，再恢复房间态。

### 3.4 本地会话状态

- 客户端保存当前连接状态、登录状态、token、用户信息和当前房间 ID。
- 所有状态变更都通过统一会话控制器驱动，不允许散落在各个 UI 页面。

## 4. 客户端状态与本地缓存

### 4.1 核心缓存对象

- 用户会话缓存：保存当前登录态、 token、角色、最近错误。
- 房间列表缓存：保存最近浏览的房间列表和分页状态。
- 房间快照缓存：保存当前房间的标题、在线人数、最近弹幕和状态。
- 未发送队列：保存网络短暂异常时需要重试的消息。

### 4.2 数据结构建议

- 会话映射建议使用 `std::unordered_map` 或轻量状态对象。
- 房间列表建议维护分页缓存和按更新时间排序的快照。
- 最近弹幕建议使用固定容量队列，防止 UI 渲染压力过大。

### 4.3 缓存原则

- 本地缓存只用于展示和恢复，不作为最终权威数据。
- 所有缓存都必须有失效策略。
- 登录切换用户时必须清理上一用户残留缓存。

## 5. 客户端接口统一标准

### 5.1 推荐接口风格

- 模块接口统一返回 `Result<T>`。
- 回调参数必须稳定，状态变化通过事件或观察者推送。
- 所有模块对外只暴露稳定头文件，不暴露内部实现。

### 5.2 核心接口建议

- `connectServer()`：建立连接。
- `login()`：执行登录。
- `logout()`：退出登录。
- `joinRoom()`：进入房间。
- `leaveRoom()`：离开房间。
- `sendDanmaku()`：发送弹幕。
- `refreshRoomList()`：刷新房间列表。
- `reconnect()`：断线重连。

### 5.3 事件通知

- 登录成功、登录失败、断线重连成功、房间切换、弹幕发送成功、弹幕发送失败都应有统一事件通知。
- UI 层仅订阅事件，不直接轮询底层状态。

## 6. 客户端弹幕模块设计

### 6.1 目标

弹幕模块负责输入、预览、发送、回显和失败重试，是客户端和服务端交互最频繁的链路之一。

### 6.2 输入与发送流程

1. 用户在输入框编辑弹幕。
2. 客户端先做长度、空值和本地预检。
3. 通过协议层封装为 `MSG_DANMAKU`。
4. 网络层发送到服务端。
5. 客户端根据服务端响应或广播回包更新展示状态。

### 6.3 展示策略

- 本地输入的弹幕可先做乐观回显，但必须和服务端确认结果对齐。
- 未通过服务端鉴权或限流的消息必须回滚状态。
- 最近弹幕展示建议采用固定容量列表。

### 6.4 失败处理

- 网络中断时保存待重试消息。
- 超时未确认时提示用户重试。
- 敏感词命中、权限不足、限流等错误必须显示可读提示。

## 7. 客户端房间模块设计

### 7.1 目标

房间模块负责房间列表、进房退房、房间状态展示、在线人数展示和房间信息刷新。

### 7.2 房间列表

- 支持分页、排序和条件筛选。
- 支持最近访问房间的本地缓存。
- 支持房间信息按更新时间刷新。

### 7.3 房间状态同步

- 客户端应根据服务端推送更新房间状态。
- 房间关闭、下播或踢出后应立即刷新页面状态。
- 在线人数以服务端同步值为准，本地仅做展示缓存。

### 7.4 进房与退房

- 进入房间前必须确认登录态有效。
- 进房成功后刷新当前房间快照。
- 退房时清理当前房间相关缓存和输入状态。

## 8. 客户端本地敏感词预检

### 8.1 定位

客户端不承担最终过滤职责，最终过滤仍由服务端完成。客户端这里只做输入辅助和提前提示。

### 8.2 预检策略

- 可选加载一份轻量词库用于本地提示。
- 命中后提示用户修改内容，但不作为最终放行依据。
- 词库更新不应阻塞主界面。

### 8.3 约束

- 客户端预检结果不得替代服务端结果。
- 预检逻辑必须可关闭，避免与服务端策略冲突。

## 9. 客户端构建与部署

### 9.1 构建目标

客户端构建应支持开发、测试和发布三种模式，确保依赖可重复、资源可打包、配置可外置。

### 9.2 推荐内容

- 保持与服务端相同的 C++17 基线。
- 网络库、序列化库和日志库版本与服务端尽量对齐。
- 配置文件中保存服务器地址、端口、重连参数和本地缓存策略。

### 9.3 部署原则

- 不把服务器地址和测试账号硬编码到程序中。
- 支持通过配置文件或启动参数切换环境。
- 打包时包含必要资源和默认配置模板。

## 10. 客户端性能优化方案

### 10.1 优化目标

客户端优化重点是首屏速度、列表滚动、消息收发稳定性和低延迟交互。

### 10.2 优化方向

- 房间列表虚拟化，避免一次性渲染过多条目。
- 弹幕展示批量刷新，避免高频重绘。
- 网络断线重连采用退避策略。
- 本地缓存采用固定容量和过期淘汰。

### 10.3 约束

- 优化不能破坏协议一致性。
- 优化不能以牺牲可维护性为代价。
- 性能收益必须能通过压测和回归测试验证。

## 11. 客户端测试方案

### 11.1 测试范围

- 协议编解码。
- 登录与断线重连。
- 房间切换与状态同步。
- 弹幕发送、回显和失败重试。
- 本地缓存清理与恢复。

### 11.2 测试层次

- 单元测试：协议解析、状态机、缓存和重试策略。
- 集成测试：连接服务端、登录、进房、发弹幕、退房。
- 联调测试：与服务端一起验证消息流转和错误处理。
- 压测辅助：模拟并发连接和高频弹幕发送。

### 11.3 质量门槛

- 所有核心路径都要有回归用例。
- 登录、进房、发弹幕、断线重连四条主链路必须稳定可测。

## 12. 客户端后期扩展方案

### 12.1 可扩展方向

- 多房间标签页。
- 礼物面板和礼物动画。
- 私信和系统通知。
- 举报、拉黑和用户管理。
- 多端登录状态同步。

### 12.2 扩展原则

- 新能力必须先落协议，再落客户端模块，再落服务端实现。
- 不允许客户端先做私有协议分支。
- 所有扩展都必须兼容旧版本服务器。

---

## 13. 交付约束

- 客户端必须优先服从服务端协议和状态机。
- 不允许客户端私自定义消息格式或字段含义。
- 所有新增能力都要在对应服务端章节找到明确映射。
- 客户端实现完成后，应能按服务端 00~12 文档逐项验收。
