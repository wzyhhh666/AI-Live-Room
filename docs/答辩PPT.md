## Slide 4: 核心功能模块（一）

| 模块 | 技术实现 | 核心亮点 |
|:----:|---------|---------|
| **实时推流** | WebRTC WHIP + H.264编码优先 | ICE连接检测，15秒超时自动重试 |
| **低延迟播放** | mpegts.js FLV播放 + 原生HLS降级 | 自动降级策略，端到端延迟 <3秒 |
| **弹幕系统** | Socket.IO 实时推送 + Canvas渲染 | 敏感词过滤（523词库），防重复发送 |
| **礼物系统** | Redis ZSET排行榜 + MySQL持久化 | 实时排行更新，6种全屏特效动画 |

### 🔧 模块详解

#### 实时推流
- 使用 **WHIP (WebRTC-HTTP Ingestion Protocol)** 标准协议
- 通过 `RTCRtpSender.setCodecPreferences()` 强制 **H.264** 编码优先
- 内置 **ICE连接状态检测**，15秒无响应自动断开重连

#### 低延迟播放
- 首选 **mpegts.js** 实现 FLV over HTTP 播放（最低延迟）
- 自动降级至 **原生 HLS** （兼容性兜底）
- 支持播放器状态管理：连接中 / 已连接 / 断开 / 错误

#### 弹幕系统
- **Socket.IO** 房间级广播，仅推送当前房间弹幕
- **Canvas** 高性能渲染，支持滚动速度调节
- **敏感词过滤器**：前缀树（Trie）匹配，523个关键词
- **防重复机制**：同一用户3秒内不可发送相同内容

#### 礼物系统
- **6种礼物类型**：🌹玫瑰、🚀火箭、💎钻石、🎵音符、❤️爱心、👑皇冠
- **Redis ZSET** 实时维护房间内礼物排行榜
- **MySQL** 异步持久化所有礼物记录
- **全屏特效动画**：CSS Keyframe + Vue Transition

## Slide 6: 关键技术难点与解决方案

### ⚠️ 难点一：VP8→H.264 转码导致视频丢失

**问题描述：**
> WebRTC 默认协商 VP8 编码，SRS 对 VP8 支持不完善，导致观众端无法正常播放视频。

**解决方案：**

```javascript
// 强制使用 H.264 编码
const codecs = RTCRtpSender.getCapabilities('video').codecs;
const h264Codecs = codecs.filter(c => c.mimeType === 'video/H264');
if (h264Codecs.length > 0) {
  sender.setCodecPreferences(h264Codecs);
}
```

**效果：** 推流成功率从 ~60% 提升至 **~98%**

---

### ⚠️ 难点二：Socket.IO 竞态条件

**问题描述：**
> 客户端在 `join-room` 事件发出后、`on()` 监听器注册前收到服务端消息，导致消息丢失。

**解决方案：**

```javascript
// ❌ 错误顺序：先join再注册监听
socket.emit('join-room', roomId);
socket.on('danmaku', handler);  // 可能丢失消息

// ✅ 正确顺序：先注册所有监听器再join
socket.on('danmaku', handler1);
socket.on('gift', handler2);
socket.on('like', handler3);
socket.emit('join-room', roomId);  // 最后才加入房间
```

---

### ⚠️ 难点三：礼物排行榜不同步

**问题描述：**
> 多人同时送礼物时，Redis 排行榜与前端显示不一致。

**解决方案：三步原子操作**

```
步骤1: Redis ZINCRBY room:{id}:gifts {amount} {userId}
          ↓
步骤2: Redis ZREVRANGE room:{id}:gifts 0 4  (获取Top5)
          ↓
步骤3: socket.to(roomId).emit('gift-rank-update', rankData)
```

通过 Redis 单线程特性保证步骤1和2的原子性。

---

### ⚠️ 难点四：刷新后房间消失

**问题描述：**
> 主播刷新页面后，房间状态未正确恢复，观众端看不到直播间。

**解决方案：**

1. **`handleUnpublish` 增加活跃流检查**：确认流真正停止后才标记房间下线
2. **Server 启动时清理 stale rooms**：遍历所有 `status='live'` 的房间，检查 SRS 是否有对应活跃流，无则重置为 `offline`

---

### ⚠️ 难点五：主播误给自己送礼

**问题描述：**
> 主播在自己的直播间中也能看到礼物面板并给自己送礼，造成数据异常。

**解决方案：**

```vue
<!-- RoomView.vue -->
<GiftPanel :canSend="!isViewerHost" />

<script setup>
// 检测当前用户是否为该房间的主播
const isViewerHost = computed(() => {
  return userStore.userId === roomStore.currentRoom?.owner_id
})
</script>
```

## Slide 11: 不足与改进方向

### ⚠️ 当前不足

| # | 不足项 | 影响程度 | 说明 |
|---|--------|---------|------|
| 1 | **缺少用户认证体系** | 🔴 高 | 当前仅用昵称标识身份，无Token/JWT安全机制 |
| 2 | **无视频录制/回放** | 🟡 中 | 直播结束后内容无法保存和回放 |
| 3 | **未实现PK连麦/多人连麦** | 🟡 中 | 仅支持单人直播，无多人互动场景 |
| 4 | **移动端适配不完善** | 🟡 中 | 响应式布局基础具备，但触控体验待优化 |
| 5 | **无负载均衡/集群方案** | 🟠 中高 | 单机部署，无法水平扩展支撑大规模并发 |

### 🚀 改进方向

#### 短期改进（1-2个月）

| 方向 | 技术方案 | 预期效果 |
|------|---------|---------|
| **认证体系** | 引入 JWT + Refresh Token + OAuth第三方登录 | 用户身份安全可信 |
| **视频录制** | 集成 FFmpeg 对 RTMP 流进行录制 | 支持直播回放功能 |
| **移动端优化** | 响应式布局重构 + PWA + Touch手势优化 | 移动端完整可用 |

#### 中长期规划（3-6个月）

| 方向 | 技术方案 | 预期效果 |
|------|---------|---------|
| **SFU多人连麦** | 引入 SFU (Selective Forwarding Unit) 服务端 | 支持PK/多人连麦场景 |
| **集群化部署** | Kubernetes 编排 + Nginx负载均衡 + Redis Cluster | 支撑万级并发 |
| **CDN加速** | 对象存储(OSS) + CDN节点分发 | 全球低延迟访问 |
| **AI增强功能** | AI弹幕审核、智能推荐、虚拟主播 | 平台智能化升级 |
