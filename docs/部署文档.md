### 2.3 Docker 基础设施启动

本项目使用 Docker Compose 编排以下基础设施服务。**在启动前后端代码之前，必须先确保这些服务已正常运行**。

#### 启动命令

```bash
docker compose -f docker/docker-compose.yml up -d mysql redis srs-server
```

> **说明**：仅启动 `mysql`、`redis`、`srs-server` 三个基础服务，不包含 `nginx` 和 `chatroom-server`（C++ 开发容器）。开发模式下前后端代码直接在本地运行。

#### 各服务详细说明

##### MySQL 8.0 数据库服务

| 配置项 | 值 | 说明 |
|--------|-----|------|
| 镜像 | `mysql:8.0` | - |
| 容器名 | `chatroom-mysql` | - |
| 映射端口 | `3309:3306` | 主机 3309 → 容器 3306 |
| Root 密码 | `root123` | - |
| 数据库名 | `chatroom_db` | 自动创建 |
| 普通用户 | `chatroom` / `chatroom123` | - |
| 字符集 | `utf8mb4` / `utf8mb4_unicode_ci` | 支持完整 Unicode（含 emoji） |
| 数据持久化 | Volume: `mysql-data` | 容器重启数据不丢失 |

连接方式（从主机）：

```bash
# 命令行连接
docker exec -it chatroom-mysql mysql -uroot -proot123 chatroom_db

# 或通过端口映射（主机端口 3309）
mysql -h 127.0.0.1 -P 3309 -uroot -proot123 chatroom_db
```

##### Redis 7 缓存服务

| 配置项 | 值 | 说明 |
|--------|-----|------|
| 镜像 | `redis:7-alpine` | 轻量级 Alpine 镜像 |
| 容器名 | `chatroom-redis` | - |
| 映射端口 | `6379:6379` | - |
| 持久化 | AOF (`appendonly yes`) | 数据安全，断电不丢数据 |
| 数据持久化 | Volume: `redis-data` | - |

Redis 在本项目中用于：

- **礼物排行榜**：Sorted Set `rank:room:{roomId}:gift`
- **弹幕发布订阅**：Channel `danmaku:input`
- **礼物事件发布订阅**：Channel `gift:input`
- **在线人数缓存**
- **频率限制计数**

##### SRS 5 流媒体服务器

| 配置项 | 值 | 说明 |
|--------|-----|------|
| 镜像 | `ossrs/srs:5` | SRS 官方镜像 |
| 容器名 | `chatroom-srs` | - |
| 重启策略 | `unless-stopped` | 异常退出自动重启 |

SRS 端口映射一览：

| 协议 | 容器端口 | 主机映射端口 | 用途 |
|------|---------|-------------|------|
| RTMP | 1935 | 1935 | 传统 RTMP 推流（OBS 等） |
| HTTP-FLV | 8080 | 8080 | 低延迟 FLV 播放 |
| HLS | 8088 | 8088 | HLS(m3u8/ts) 播放 |
| HTTP API / WHIP | 1985 | 1985 | WebRTC 信令 + WHIP/WHEP API |
| WebRTC UDP | 8000 | 8000/udp | WebRTC 数据传输 |
| WebRTC TCP | 8000 | 8000/tcp | WebRTC TCP Candidate |

SRS 核心功能链路：

```
浏览器(主播) --WHIP--> SRS:1985 --> rtc_to_rtmp --> RTMP内部转换
                                              |
                    浏览器(观众) <--FLV-- SRS:8080 <-- http_remux
                                              |
                    浏览片(观众) <--HLS-- SRS:8088 <-- hls切片
                                              |
                    浏览器(观众) <--WebRTC-- SRS:8000 <-- rtc
```

### 2.5 配置环境变量

#### 后端环境变量（api-server/.env）

在 `api-server/` 目录下创建 `.env` 文件（如果不存在）：

```env
# API Server
API_PORT=3000
NODE_ENV=development

# MySQL（注意端口是 3309，不是默认的 3306）
MYSQL_HOST=127.0.0.1
MYSQL_PORT=3309
MYSQL_USER=root
MYSQL_PASSWORD=root123
MYSQL_DATABASE=chatroom_db

# Redis
REDIS_HOST=127.0.0.1
REDIS_PORT=6379
REDIS_PASSWORD=

# 流媒体地址（替换为你的局域网 IP）
STREAM_HOST=YOUR_LAN_IP
API_HOST=YOUR_LAN_IP
```

> **重要**：将 `YOUR_LAN_IP` 替换为你实际的局域网 IP，例如 `10.56.132.159`。

#### 前端环境变量（frontend/.env.development）

编辑或确认 `frontend/.env.development` 内容：

```env
# SRS 流媒体地址（替换为你的局域网 IP）
VITE_SRS_URL=https://YOUR_LAN_IP
VITE_SRS_RTC_URL=https://YOUR_LAN_IP:1985
```

> **重要**：
> - `VITE_SRS_RTC_URL` 必须包含端口号 `1985`（WHIP/WebRTC 信令端口）
> - 开发环境下 Vite 的 proxy 会将 `/rtc`、`/hls`、`/live` 路径代理到对应 SRS 端口
> - 使用 `https://` 是因为 WebRTC WHIP 推流需要安全上下文（localhost 例外）

#### CORS 白名单配置

[api-server/config/database.js](api-server/config/database.js) 中的 `cors.origin` 已预置了常见地址。如果你使用不同的 IP 或端口，需修改该文件：

```javascript
cors: {
  origin: [
    'http://localhost:5173',
    'http://YOUR_LAN_IP:5173',    // ← 添加你的局域网地址
    'https://YOUR_LAN_IP',        // ← 添加 HTTPS 地址
  ],
  credentials: true
}
```

### 3.2 Nginx 反向代理配置

Nginx 作为统一入口，负责路由分发和 SSL 终结。配置文件位于 [docker/nginx/nginx.conf](docker/nginx/nginx.conf)。

#### 路由规则详解

```
请求路径                          →  转发目标
─────────────────────────────────────────────────
/api/*                           →  api_server (host.docker.internal:3000)
/socket.io/*                     →  api_server (WebSocket 升级支持)
/hls/*                           →  srs_hls (srs-server:8088)
/live/*                          →  srs_flv (srs-server:8080)
/health                          →  api_server
/*                               →  api_server (兜底)
```

#### Upstream 定义

```nginx
upstream api_server {
    server host.docker.internal:3000;   # 主机上的 API 服务
}

upstream srs_hls {
    server srs-server:8088;             # Docker 网络内的 SRS HLS
}

upstream srs_flv {
    server srs-server:8080;             # Docker 网络内的 SRS FLV
}
```

#### 关键配置要点

1. **HTTP → HTTPS 自动跳转**：80 端口收到请求后 301 重定向到 HTTPS
2. **WebSocket 支持**：`/socket.io/` 路径配置了 `Upgrade` 和 `Connection` 头
3. **真实 IP 传递**：通过 `X-Real-IP` 和 `X-Forwarded-For` 头传递客户端 IP
4. **Docker 网络互通**：Nginx 容器通过 `host.docker.internal` 访问宿主机服务，通过服务名访问同网络内其他容器

#### Nginx Dockerfile 说明

[docker/nginx/Dockerfile](docker/nginx/Dockerfile) 基于 `nginx:1.25-alpine` 构建：

```dockerfile
FROM nginx:1.25-alpine
RUN apk add --no-cache openssl
RUN mkdir -p /etc/nginx/ssl && \
    openssl req -x509 -nodes -days 365 -newkey rsa:2048 \
      -keyout /etc/nginx/ssl/key.pem \
      -out /etc/nginx/ssl/cert.pem \
      -subj "/C=CN/ST=Beijing/L=Beijing/O=Chatroom/CN=localhost"
COPY nginx.conf /etc/nginx/nginx.conf
```

> **注意**：当前使用自签名证书（仅用于开发/测试）。生产环境请替换为正式 CA 签发的证书（见 3.3 节）。

### 3.3 SSL/HTTPS 配置说明

#### 为什么需要 HTTPS

WebRTC WHIP 推流协议要求**安全上下文**（Secure Context），即：
- `https://` 地址
- 或 `localhost` / `127.0.0.1`（例外）

因此即使在内网部署，也建议配置 HTTPS。

#### 方案 A：自签名证书（开发/测试用）

Docker 构建时已自动生成自签名证书（有效期 365 天），位于容器内 `/etc/nginx/ssl/` 目录。

浏览器访问时会提示「不安全」，需手动点击「继续访问」。

#### 方案 B：Let's Encrypt 免费证书（推荐生产使用）

```bash
# 1. 安装 certbot
sudo apt install certbot python3-certbot-nginx   # Ubuntu
# 或 Windows 上使用 win-acme 等工具

# 2. 申请证书（假设域名是 live.example.com）
sudo certbot --nginx -d live.example.com

# 3. 将证书挂载到 Nginx 容器
# 修改 docker-compose.yml 的 nginx 服务：
nginx:
  volumes:
    - /etc/letsencrypt/live/live.example.com/cert.pem:/etc/nginx/ssl/cert.pem:ro
    - /etc/letsencrypt/live/live.example.com/privkey.pem:/etc/nginx/ssl/key.pem:ro
```

#### 方案 C：自有证书

**排查步骤**：

```bash

docker compose -f docker/docker-compose.yml logs mysql
```

**Windows 特有问题**：

- 确保 WSL2 后端已启用：Docker Desktop → Settings → General → Use the WSL 2 based engine
- 确保 Docker Desktop 服务正在运行（系统托盘图标可见）

| .env 中 MYSQL_PORT 错误 | 确认值为 `3309`（不是 `3306`） |
| Windows 防火墙拦截 | 添加入站规则允许 3309 端口 |

**验证 MySQL 可达性**：


## 6. 性能优化建议

### 6.1 数据库索引策略

当前 [01_init.sql](docker/init-sql/01_init.sql) 中已有的索引：

```sql
-- 弹幕消息表：按房间+时间查询
INDEX idx_room_created (room_id, created_at)

-- 礼物记录表：按房间+时间查询
INDEX idx_room_time (room_id, created_at)

-- 房间成员表：按房间+加入时间查询
INDEX idx_room_join (room_id, join_time)

-- 房间成员表：唯一约束防止重复加入
UNIQUE KEY uk_room_user (room_id, user_id)
```

**建议补充的索引**：

```sql
-- 用户表：按 username 登录查询
ALTER TABLE users ADD INDEX idx_username (username);

-- 直播间表：按 host_id 查询主播的所有房间
ALTER TABLE rooms ADD INDEX idx_host_id (host_id);

-- 直播间表：按 state 筛选正在直播的房间
ALTER TABLE rooms ADD INDEX idx_state (state);

-- 礼物记录表：按 sender_id 查询用户的送礼历史
ALTER TABLE gift_record ADD INDEX idx_sender (sender_id, created_at);
```

### 6.2 Redis 缓存层

当前 Redis 使用场景及优化建议：

| 场景 | 当前实现 | 优化建议 |
|------|---------|---------|
| 礼物排行榜 | Sorted Set（持久化） | ✅ 已合理，无需改动 |
| 弹幕 PubSub | Publish/Subscribe | ✅ 已合理 |
| 房间列表 | 每次查 MySQL | 📈 增加 5s TTL 缓存 |
| 热门直播间 | 无 | 📈 使用 Sorted Set 按热度排序，TTL 30s |
| 用户 Token 校验 | 无缓存 | 📈 Token → UserID 映射缓存，TTL 与 Token 有效期一致 |
| 在线人数 | Socket.io 内存 | ✅ 已合理（实时准确） |

**添加房间列表缓存的示例**：

```javascript
// 在 roomService.getRoomList() 中
const cacheKey = 'rooms:list:page:' + page;
const cached = await redis.get(cacheKey);
if (cached) return JSON.parse(cached);

const result = await queryFromMySQL();
await redis.setex(cacheKey, 5, JSON.stringify(result)); // 5 秒 TTL
return result;
```

**Nginx 静态资源缓存配置**（追加到 nginx.conf）：

```nginx
# 静态资源长期缓存
location ~* \.(js|css|png|jpg|jpeg|gif|ico|svg|woff2?)$ {
    expires 30d;
    add_header Cache-Control "public, immutable";
}

# HTML 不缓存
location ~* \.html$ {
    expires -1;
    add_header Cache-Control "no-store, no-cache, must-revalidate";
}
```

**Gzip 压缩**（追加到 nginx.conf 的 http 块）：

```gzip
gzip on;
gzip_vary on;
gzip_min_length 1024;
gzip_types text/plain text/css application/json application/javascript text/xml image/svg+xml;
```

### 6.5 SRS 性能参数调优

根据 [srs/srs.conf](srs/srs.conf)，可根据实际情况调整：

| 参数 | 当前值 | 建议（高并发） | 说明 |
|------|-------|--------------|------|
| `max_connections` | 1000 | 5000+ | 最大并发连接数 |
| `hls_fragment` | 3 | 2 | HLS 分片长度（秒），越短延迟越低 |
| `hls_window` | 15 | 30 | HLS 窗口长度（秒），影响回看范围 |
| `worker_connections`(Nginx) | 2048 | 4096+ | Nginx 并发连接数 |
