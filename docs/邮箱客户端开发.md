# 邮箱客户端开发（直连模式，Qt 5.12.11）— 开发流程与顺序

> 生成日期：2026-05-30

## 0. 目标与边界（先定清楚，避免走偏）

### 0.1 你要做的是什么
做一个**邮件客户端（Mail Client）**，直接与真实邮箱服务商（QQ/163 等）的 **SMTP（发信）** + **IMAP（收信）** 通信：
- 发送邮件（text/plain + 附件）
- 登录认证（优先使用“授权码/应用专用密码”）
- 收取邮件（列表 + 详情）
- 邮件格式解析（RFC5322 + MIME）
- 附件 Base64/QP 编解码
- 纯文本邮件渲染
- 多账号管理
- 邮件列表与详情查看
- 草稿箱/发件箱本地持久化（SQLite 或 MySQL）

### 0.2 你不需要做什么
- 不需要自建公网邮件服务器（不需要 SPF / DKIM / DMARC / 反向 DNS / 投递信誉等）
- 不需要对外开放 25 端口

### 0.3 重要前提（QQ/163 直连的现实约束）
- 通常要在网页邮箱设置里**开启 IMAP/SMTP**
- 通常需要使用“**授权码/应用专用密码**”替代登录密码
- 服务器地址/端口/SSL 方式不同服务商可能变化：以官方说明为准

## 1. 技术栈建议（贴合你已掌握的能力）

### 1.1 核心框架
- UI：Qt Widgets 或 QML（建议先 Widgets，快）
- 网络：Qt `QTcpSocket/QSslSocket`（事件驱动，天然适配交互式协议）
- 解析：你自己实现最小 RFC5322/MIME 子集（练核心能力）
- 本地存储：默认 **SQLite**（单机最省事）；MySQL 仅在你有“本机/局域网 DB”需求时再上
- 加密：本地账号凭据加密（AES-GCM + KDF），你正好会 AES/RSA

### 1.2 Windows + Qt 5.12.11 的 TLS 提示
Qt 的 `QSslSocket` 在 Windows 上依赖 OpenSSL 动态库（版本需与 Qt 匹配）。开发期务必先做一次自检：
- `QSslSocket::supportsSsl()`
- `QSslSocket::sslLibraryBuildVersionString()` / `sslLibraryVersionString()`

### 1.3 你自写的 `TcpClient` vs `QTcpSocket/QSslSocket`（建议结论）

#### 建议结论
- 做 IMAP/SMTP/POP3 这种“标准邮箱协议”的直连客户端：**优先用 Qt 的 `QSslSocket`**（必要时配合 STARTTLS）。
- 你网盘项目里那类“先收 4 字节长度再收包体”的 `TcpClient`：更适合**自定义二进制协议**，不适合直接拿来连 QQ/163 的 IMAP/SMTP。

#### 关键区别（为什么不适合直接复用你的 `TcpClient`）
- **协议形态不同**：SMTP/IMAP/POP3 是 **CRLF 文本协议**（多行响应、`.` 结束符、IMAP literal `{n}`），不是长度前缀包。
- **TLS 必需**：QQ/163 的 IMAP/SMTP 基本都要求 TLS（隐式 TLS 或 STARTTLS）。你 `TcpClient` 需要额外集成 OpenSSL 并做完整握手/证书校验；这等于重复造 `QSslSocket`。
- **Qt 集成度**：`QSslSocket`/`QTcpSocket` 天然适配 Qt 事件循环与信号槽，UI 工程里更稳定、更省代码。

#### STARTTLS vs “SSL” vs TLS（你提到的疑问）
- **TLS 是 SSL 的后继者**：SSLv2/SSLv3 已淘汰、不安全；现代邮箱服务用的是 TLS（通常 TLS 1.2/1.3）。
- Qt API 名字里叫 `QSslSocket` 是历史命名，不代表你在用过时的 SSL 协议。
- 常见两种“用 TLS 的方式”：
  - **隐式 TLS**：一连接就 TLS（IMAPS 993、SMTPS 465 常见）
  - **STARTTLS**：先明文连接，再发 `STARTTLS`/`STLS` 升级为 TLS（SMTP 587、IMAP 143 常见）

> 实战建议：用 `QSslConfiguration` 把协议限制为 `TlsV1_2OrLater`，避免被协商到旧版本。

## 2. 推荐架构（直连模式的最小分层）

### 2.1 分层
- **UI 层**：账号列表、邮箱列表、详情、写邮件、附件选择
- **Domain 层**：Account、Message、Attachment、Draft、OutboxItem
- **Protocol 层**：
  - `SmtpClient`（SMTP submission：465/587 常见）
  - `ImapClient`（IMAPS：993 常见）
  - `TlsSocket`（对 `QSslSocket` 的薄封装：readLine/readBytes/write、超时）
- **MIME 层**：
  - `Rfc5322HeaderParser`
  - `MimeParser`（multipart tree）
  - `TransferCodec`（Base64/QP）
  - `Composer`（发信组包：text/plain + multipart/mixed）
- **Storage 层**：SQLiteRepository / MySqlRepository（接口一致，底层可切换）

### 2.2 并发模型（Qt 里别硬套 epoll）
- 网络 I/O：Qt 信号槽（主线程事件循环）
- CPU/解析/解码：可丢到线程池（`QThreadPool` 或你自带线程池）
- 原则：**协议读写不要在 UI 线程阻塞**；解析可以异步

### 2.3 单一客户端如何做多账号管理（直连模式）

“单一客户端”指的是**同一个程序**支持多个邮箱账号，并不是“同一时间只能登录一个账号”。多账号管理的关键在于：
- **账号配置集中存储**：用 `accounts` 表保存每个账号的 IMAP/SMTP 配置与凭据（加密后）
- **所有数据按 account_id 隔离**：列表缓存、原文、附件、草稿、Outbox、Sent 都必须带 `account_id`
- **连接/会话按账号维护**：每个账号逻辑上都有自己的 `ImapClient`/`SmtpClient` 状态

#### 最小实现（推荐先做这个，最省心）
- UI 提供“账号下拉框/侧边栏账号列表”
- 客户端只维持 **当前选中账号**的连接：
  - 切换账号：断开旧 IMAP（或让它空闲超时断开）-> 连接新 IMAP -> 从本地缓存加载列表 -> 用户手动点“同步”刷新
  - SMTP 发信：按当前账号配置临时建立连接发送（或复用一个短会话）

这种做法已经满足：
- 一个客户端管理多个账号
- 账号切换后看到不同 Inbox 列表/草稿/Outbox（因为数据按 account_id 隔离）

#### 进阶实现（等最小闭环跑通再做）
- 多账号同时后台同步：为每个账号建立独立的同步任务（可限并发，比如一次最多同步 1~2 个账号）
- 为每个账号维护“上次同步的 UID/时间戳”，做增量拉取
- UI 显示每个账号的未读数/同步状态

#### 常见踩坑（直连多账号）
- 只在表里加 `account_id` 不够：所有查询、主键/唯一键都要把 `account_id` 纳入条件（避免不同账号 UID 冲突）
- 不要把“当前账号”写死成全局单例字符串：用 `account_id` 作为全局上下文更稳
- 凭据不要明文存储：至少做到 `secret_enc`（AES-GCM + KDF）

## 3. 协议最小实现（你要支持的“最小子集”）

### 3.1 SMTP（发信）最小子集
目标：先发一封**纯文本无附件**邮件给自己。

必须支持：
- `EHLO`
- `STARTTLS`（若使用 587）或直接 TLS（465）
- `AUTH`（常见 LOGIN/PLAIN；不同服务商不同）
- `MAIL FROM` / `RCPT TO` / `DATA` / `QUIT`

关键坑：
- 全程 `\r\n`
- `DATA` 结束必须 `\r\n.\r\n`
- dot-stuffing：正文中以 `.` 开头的行要前置一个 `.`

### 3.2 IMAP（收信）最小子集
目标：先做 INBOX 列表（headers），再做详情（全文）。

必须支持：
- `LOGIN`（或更复杂的 SASL/OAuth2：先不做）
- `SELECT INBOX`
- `UID SEARCH`（用 UID 做增量）
- `UID FETCH` headers（先拉列表字段）
- `UID FETCH BODY.PEEK[]`（点开详情再拉全文）
- `LOGOUT`

关键坑：
- tagged/untagged response
- literal `{n}`：必须读 n 字节（可能包含任意字节）

## 4. MIME/附件：最小可用解析范围

### 4.1 先支持的 MIME 子集
- `text/plain; charset=...`
- `multipart/mixed; boundary=...`（附件）
- `Content-Transfer-Encoding`：`base64`、`quoted-printable`、`7bit/8bit`
- `Content-Disposition: attachment; filename=...`（先支持最常见形态）

### 4.2 渲染（只做纯文本）
- 转码到 UTF-8（建议 ICU/iconv；否则容易被 charset 坑死）
- 换行统一：`\r\n`/`\r` -> `\n`
- QP 解码后再渲染（注意软换行 `=`）

## 5. 本地持久化（SQLite 优先；MySQL 可选）

### 5.1 为什么 SQLite 更适合直连客户端
- 单机、零部署、一个文件即可
- 适合缓存列表/详情、草稿、Outbox、Sent

MySQL 适用场景：你想把本地数据放到局域网 DB、或你习惯 MySQL 工具链。

### 5.2 最小表设计（SQLite/MySQL 通用思路）
> 表名与字段保持通用命名，便于后续迁移。

- `accounts`
  - `id`, `email`, `display_name`, `imap_host`, `imap_port`, `imap_tls_mode`, `smtp_host`, `smtp_port`, `smtp_tls_mode`, `auth_type`, `username`, `secret_enc`, `created_at`
- `mailbox_cache`
  - `account_id`, `mailbox`, `uid`, `flags`, `from_addr`, `subject`, `date_raw`, `snippet`, `size_bytes`, `updated_at`
- `message_raw`（可选：存 raw 或文件路径）
  - `account_id`, `uid`, `raw_path` 或 `raw_blob`, `fetched_at`
- `attachments`
  - `id`, `account_id`, `uid`, `filename`, `mime_type`, `size_bytes`, `saved_path`, `created_at`
- `drafts`
  - `id`, `account_id`, `to_json`, `cc_json`, `subject`, `body_text`, `attachments_json`, `updated_at`
- `outbox`
  - `id`, `account_id`, `mime_path`, `status`, `retry_count`, `last_error`, `updated_at`
- `sent`
  - `id`, `account_id`, `to_json`, `subject`, `date_sent`, `mime_path`

### 5.3 本地凭据加密（强烈建议做）
- 方案：主密码 -> PBKDF2/Argon2 派生 key -> AES-GCM 加密 `secret_enc`
- 你会 RSA：可以不必用在本地加密上（会增加复杂度），先把 AES-GCM/KDF 做稳

## 6. 开发流程与顺序（最小化直连版，按里程碑推进）

> 目标：每 1-2 天都有可验收成果。

### 阶段 A：工程骨架（0.5~1 天）
1) 创建 Qt 工程（Widgets）
2) 增加基础模块目录：`protocol/ mime/ storage/ domain/ ui/`
3) 日志与配置（先用 ini/json 都行）

验收：能启动、能保存/读取配置。

### 阶段 B：TLS 连接自检（0.5 天）
1) 写一个小工具页：输入 host/port，尝试 TLS 连接
2) 打印 `supportsSsl` 与 OpenSSL 版本信息

验收：能成功建立 IMAPS/SMTPS 连接（至少握手成功）。

### 阶段 C：SMTP 发送纯文本（1~2 天）
1) `SmtpClient`：实现 `EHLO -> AUTH -> MAIL/RCPT/DATA -> QUIT`
2) `Composer`：先只生成最简单 RFC5322：From/To/Subject/Date + 空行 + body
3) UI：写信页（To/Subject/Body）

验收：你在网页邮箱里能收到自己发的邮件。

### 阶段 D：IMAP 拉 INBOX 列表（1~3 天）
1) `ImapClient`：实现 `LOGIN -> SELECT INBOX -> UID SEARCH -> UID FETCH headers`
2) 把 headers（from/subject/date/flags/size）写入 `mailbox_cache`
3) UI：列表页显示 50 封

验收：列表能稳定刷新、重启后缓存仍在。

### 阶段 E：IMAP 拉详情 + 纯文本渲染（2~4 天）
1) 点开邮件：`UID FETCH BODY.PEEK[]` 拉原文
2) `Rfc5322HeaderParser` + 最小 `MimeParser`：找到 `text/plain`
3) 转码到 UTF-8 并显示

验收：中文主题/昵称不一定完美，但正文能正确显示（至少 UTF-8 邮件正确）。

### 阶段 F：附件（2~4 天）
1) MIME 解析 multipart，定位 attachment part
2) Base64/QP 解码后落盘，写 `attachments`
3) UI：附件列表 + 保存按钮

验收：附件可保存且可正常打开。

### 阶段 G：多账号 + 草稿/Outbox/Sent（2~5 天）
1) `AccountManager`：增删改查账号、切换当前账号
2) 草稿：自动保存到 `drafts`
3) Outbox：发送失败入队重试；成功写入 `sent`

验收：断网/重启后草稿与 Outbox 状态不丢。

## 7. 你现有“高性能技术栈”如何在直连客户端里发挥价值

- 线程池/生产者消费者：用于“解析/解码/落库/索引”异步化，避免卡 UI
- 内存池：可用于 MIME 解析/附件解码缓冲区复用（先别过早优化）
- Base64：附件与 AUTH 认证都会用到
- AES/RSA：账号凭据本地加密（AES-GCM + KDF）
- SQLite：草稿/Outbox/缓存列表的核心
- 索引：SQLite 索引（按 account_id+uid、date 排序）即可覆盖主要查询

## 8. 验收清单（最终你应当达到的效果）

- 能添加 QQ/163 账号（授权码）并通过 IMAP 拉到 INBOX 列表
- 能点开邮件显示 `text/plain` 正文
- 能下载附件并保存
- 能写邮件并通过 SMTP 发出
- 支持多个账号切换
- 草稿箱/发件箱（Outbox/Sent）本地持久化，重启不丢
