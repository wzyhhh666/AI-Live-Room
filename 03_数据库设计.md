# C++ 直播弹幕+房间系统 — 数据库设计

## 1. 设计概述

本项目采用 **MySQL 8.0 + Redis 7.x** 作为数据存储组合。

- MySQL 负责持久化存储。
- Redis 负责在线状态、排行榜、房间热数据和限流。
- 表设计必须与协议、房间状态机和弹幕流程一致。

---

## 2. 核心表设计

### 2.1 users 用户表

```sql
CREATE TABLE users (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_name VARCHAR(64) NOT NULL UNIQUE,
    password_hash VARCHAR(255) NOT NULL,
    avatar_url VARCHAR(255) DEFAULT '',
    role TINYINT NOT NULL DEFAULT 0,
    status TINYINT NOT NULL DEFAULT 1,
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

### 2.2 rooms 房间表

```sql
CREATE TABLE rooms (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    room_name VARCHAR(128) NOT NULL,
    host_user_id BIGINT NOT NULL,
    cover_url VARCHAR(255) DEFAULT '',
    room_state TINYINT NOT NULL DEFAULT 0,
    online_count INT NOT NULL DEFAULT 0,
    danmaku_count BIGINT NOT NULL DEFAULT 0,
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_host_user_id (host_user_id),
    INDEX idx_room_state (room_state)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

### 2.3 room_members 房间成员表

```sql
CREATE TABLE room_members (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    room_id BIGINT NOT NULL,
    user_id BIGINT NOT NULL,
    member_role TINYINT NOT NULL DEFAULT 0,
    join_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    leave_time DATETIME DEFAULT NULL,
    UNIQUE KEY uk_room_user (room_id, user_id),
    INDEX idx_room_id (room_id),
    INDEX idx_user_id (user_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

### 2.3.1 写入策略

- `uk_room_user (room_id, user_id)` 表示同一房间、同一用户在当日只保留一条成员记录。
- 用户首次进入房间时执行普通插入。
- 用户离开房间时只更新同一行的 `leave_time`。
- 用户当天再次进入同一房间时，必须使用 `INSERT ... ON DUPLICATE KEY UPDATE` 复用原记录，更新 `join_time` 并清空 `leave_time`。
- 不要为“已离开记录保留历史”而移除唯一约束；历史行为仍由同一行上的时间字段表达。

示例：

```sql
INSERT INTO room_members (room_id, user_id, member_role, join_time, leave_time)
VALUES (?, ?, ?, NOW(), NULL)
ON DUPLICATE KEY UPDATE
    member_role = VALUES(member_role),
    join_time = VALUES(join_time),
    leave_time = NULL;
```

### 2.4 danmaku_messages 弹幕消息表

```sql
CREATE TABLE danmaku_messages (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    room_id BIGINT NOT NULL,
    user_id BIGINT NOT NULL,
    content VARCHAR(512) NOT NULL,
    content_status TINYINT NOT NULL DEFAULT 0,
    sent_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_room_sent_at (room_id, sent_at),
    INDEX idx_user_sent_at (user_id, sent_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

### 2.5 gifts 礼物记录表

```sql
CREATE TABLE gifts (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    room_id BIGINT NOT NULL,
    user_id BIGINT NOT NULL,
    gift_name VARCHAR(64) NOT NULL,
    gift_count INT NOT NULL DEFAULT 1,
    gift_value INT NOT NULL DEFAULT 0,
    sent_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_room_gift_time (room_id, sent_at),
    INDEX idx_user_gift_time (user_id, sent_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

### 2.6 sensitive_words 敏感词表

```sql
CREATE TABLE sensitive_words (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    word VARCHAR(128) NOT NULL UNIQUE,
    level TINYINT NOT NULL DEFAULT 1,
    status TINYINT NOT NULL DEFAULT 1,
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

## 3. 表关系说明

```text
users 1 ── n rooms
users 1 ── n room_members
rooms 1 ── n room_members
rooms 1 ── n danmaku_messages
rooms 1 ── n gifts
users 1 ── n danmaku_messages
users 1 ── n gifts
```

### 3.1 关系约束

- 主播用户创建房间。
- 用户加入房间后写入成员表。
- 弹幕和礼物都必须挂在具体房间下。
- 敏感词表为全局共享数据。

## 4. Redis 数据结构设计

### 4.1 房间信息

| Key | 类型 | 用途 |
|-----|------|------|
| `room:info:{roomId}` | HASH | 房间基本信息、状态、在线数 |
| `room:online:{roomId}` | SET | 当前在线用户集合 |
| `room:members:{roomId}` | SET | 当日加入过房间的成员集合 |
| `room:recent_danmaku:{roomId}` | LIST | 最近弹幕缓存 |

### 4.2 排行榜

| Key | 类型 | 用途 |
|-----|------|------|
| `rank:room:gift` | ZSET | 房间礼物榜 |
| `rank:user:score` | ZSET | 用户综合积分榜 |
| `rank:room:danmaku` | ZSET | 房间弹幕活跃度榜 |

### 4.3 在线与限流

| Key | 类型 | 用途 |
|-----|------|------|
| `online:user:{userId}` | STRING | 用户在线标记 |
| `limit:danmaku:user:{userId}` | STRING | 单用户发言限流计数 |
| `limit:danmaku:room:{roomId}` | STRING | 房间级限流计数 |

### 4.4 Redis 使用原则

- 热数据先放 Redis，冷数据落 MySQL。
- 排行榜优先使用 ZSET。
- 在线状态必须有过期时间，避免脏在线。
- 任何 Redis key 都必须有统一前缀，禁止散写。

### 4.5 语义区分

- `room:online:{roomId}`：实时在线用户集合，来源于连接建立和 JOIN_ROOM 成功；在用户连接和断开时同步更新，供广播路由和在线人数展示使用。
- `room:members:{roomId}`：房间历史成员集合，来源于 JOIN_ROOM 成功时写入；当天重复进入同一房间时复用同一条成员记录，供当日访客统计和成员去重使用。
- `room:online:{roomId}` 是判断用户“是否当前在线”的权威数据源。
- `room:members:{roomId}` 覆盖当日所有加入过房间的用户；用户离开后记录保留，重新进入时更新同一条记录。

## 5. 数据库连接池配置

### 5.1 MySQL 连接池

建议配置：

- 最小连接数：4
- 最大连接数：32
- 空闲回收：60 秒
- 连接超时：3 秒
- 读写分离预留接口：保留，但 V1 不强制实现

### 5.2 Redis 连接池

建议配置：

- 最小连接数：2
- 最大连接数：16
- 连接超时：2 秒
- 失败重试：3 次

### 5.3 访问约束

- 数据库访问只允许通过封装层进行。
- 禁止业务代码直接拼接 SQL。
- 所有查询都要带索引意识，避免全表扫描。

## 6. 初始化顺序

1. 初始化数据库连接池。
2. 初始化 Redis 连接池。
3. 创建必要表结构。
4. 加载房间缓存和排行榜。
5. 启动服务。
