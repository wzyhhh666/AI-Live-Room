# C++ 直播弹幕+房间系统 — 房间模块设计

## 1. 模块目标

房间模块负责房间的创建、状态流转、成员管理、权限控制和列表查询，是弹幕业务的承载容器。

---

## 2. 房间状态机

### 2.1 状态定义

| 状态 | 含义 |
|------|------|
| `CREATED` | 房间已创建，未开播 |
| `READY` | 房间准备完成，等待开播 |
| `LIVE` | 正在直播中 |
| `OFFLINE` | 已下播，但房间尚未关闭 |
| `CLOSED` | 房间关闭，不能继续进入 |

### 2.2 状态转移

```text
CREATED -> READY -> LIVE -> OFFLINE -> CLOSED
```

### 2.3 状态转移约束

- 创建后默认是 `CREATED`。
- 主播完成准备后切到 `READY`。
- 开播后切到 `LIVE`。
- 停播后切到 `OFFLINE`。
- 关闭后进入 `CLOSED`，不可逆回。

---

## 3. 房间生命周期管理

### 3.1 创建

- 主播发起创建请求。
- 系统生成房间 ID。
- 写入房间基础信息。
- 初始化 Redis 房间缓存。

### 3.2 开播

- 主播确认开播后切换状态。
- 房间信息对外可见。
- 开播时间写入数据库。

### 3.3 直播中

- 允许观众进入、发送弹幕、送礼物。
- 维护在线人数和活跃状态。
- 持续刷新房间热数据。

### 3.4 下播与关闭

- 下播后停止接收新互动消息。
- 关闭后清理缓存和临时状态。
- 历史数据保留在数据库中。

---

## 4. 主播-观众权限模型

### 4.1 角色定义

| 角色 | 权限 |
|------|------|
| 主播 | 创建、开播、下播、关闭、踢人、置顶消息 |
| 管理员 | 管理房间、处理违规内容、查看数据 |
| 观众 | 进入房间、发弹幕、送礼物、查看状态 |
| 游客 | 只读浏览基础房间信息 |

### 4.2 权限原则

- 进入房间前必须做权限校验。
- 弹幕和礼物权限与房间状态绑定。
- 房间关闭后只保留只读历史查看能力。

---

## 5. 房间列表查询与分页

### 5.1 查询条件

支持以下条件：

- 房间状态
- 主播 ID
- 房间名关键词
- 在线人数区间
- 热度排序

### 5.2 分页规则

- 分页参数必须统一。
- 列表默认按热度或开播时间排序。
- 返回总数、页码、每页大小和总页数。

### 5.3 查询建议

- 热门房间列表使用 Redis + MySQL 混合读。
- 管理后台列表使用数据库分页。
- 房间状态变化要同步更新缓存。

---

## 6. 房间模块接口建议

- `createRoom()`：创建房间。
- `startLive()`：开始直播。
- `stopLive()`：结束直播。
- `closeRoom()`：关闭房间。
- `joinRoom()`：用户进入房间。
- `leaveRoom()`：用户离开房间。
- `queryRoomList()`：查询房间列表。

---

## 7. 关键约束

- 房间状态机必须唯一且有序。
- 不能让未开播房间执行直播态操作。
- 不能让关闭房间继续接收弹幕。

---

## 8. 在线人数同步策略

### 8.1 数据层级

| 存储 | 字段/Key | 用途 | 更新频率 |
|------|----------|------|----------|
| 本地内存 | Connection 映射 | 实时精确在线数 | 毫秒级 |
| Redis | room:online:{roomId} | 广播路由 + 在线列表 | 毫秒级 |
| Redis | room:info:{roomId} 的 online_count | 房间信息缓存 | 1~3 秒 |
| MySQL | rooms.online_count | 持久化记录 | 每 10~30 秒批量更新 |

### 8.2 权威数据源

- 本地 Connection 映射是在线人数统计的唯一权威来源。
- Redis 与 MySQL 的 online_count 仅作为最终一致的近似值。

### 8.3 更新流程

用户进入房间：

1. 更新本地 Connection 映射。
2. Redis 执行 `SADD room:online:{roomId}`。
3. Redis 执行 `HINCRBY room:info:{roomId} online_count 1`。
4. MySQL 异步批量更新，不走主链路。

用户离开房间（正常退出）：

1. 从本地 Connection 映射移除。
2. Redis 执行 `SREM room:online:{roomId}`。
3. Redis 执行 `HINCRBY room:info:{roomId} online_count -1`。
4. MySQL 异步批量更新。

用户异常断线：

1. 定时线程检测到超时后触发断线处理。
2. 按照正常离开的流程执行上述 1~3 步。
3. 额外清理会话信息和历史在线标记。

### 8.4 MySQL 批量更新策略

- 工作线程不直接写 MySQL online_count。
- 定时线程每 10~30 秒扫描一次房间在线数变化。
- 只对 online_count 发生变化的房间执行 UPDATE。
- 房间关闭时强制同步一次 MySQL。

---

## 9. 用户异常断线处理

### 9.1 触发条件

- 心跳超时，连续多次未收到心跳。
- TCP 连接异常断开，read 返回 0 或错误。
- epoll 检测到 EPOLLERR 或 EPOLLHUP。

### 9.2 清理流程

1. 从 ConnectionManager 移除 fd 到 Connection 的映射。
2. 如果 Connection.userId 不为 0，则清理 Redis 在线状态。
3. 删除会话 token 和用户 fd 映射。
4. 异步记录用户离开时间并更新房间在线数。
5. 在 roomId 有效时广播用户离开消息。
6. 关闭 fd 并释放 Connection 对象。

### 9.3 边界情况

- 如果用户理论上关联多个房间，需要逐一清理。
- 如果用户正在发送弹幕时断线，已进入工作线程的任务仍继续完成，但不再向该 fd 发送响应。
- 如果大量连接同时断线，需要分批清理，避免瞬间写入过多 Redis 或 MySQL。
