# C++ 直播弹幕+房间系统 — 接口统一标准

## 0. 使用方式

先按照这里统一模块间接口风格，再开始拆分具体业务模块。这样可以避免函数命名、返回值和回调风格不一致。

---

## 1. 函数命名规范

### 1.1 基本规则

- 函数名必须以动词开头。
- 统一使用 camelCase。
- 公共接口尽量表达动作和结果，不写含糊名称。

### 1.2 推荐前缀

| 前缀 | 含义 | 示例 |
|------|------|------|
| handle | 处理输入事件 | `handleConnect()` |
| process | 处理业务流程 | `processDanmaku()` |
| send | 发送消息 | `sendBroadcast()` |
| load | 加载资源 | `loadConfig()` |
| save | 保存数据 | `saveRoomState()` |
| update | 更新状态 | `updateOnlineCount()` |
| query | 查询数据 | `queryRoomList()` |

### 1.3 禁止命名

- 禁止使用 `doIt()`、`func1()`、`test()` 这类无意义名字。
- 禁止一个函数名表达多个动作。

---

## 2. 返回值规范

### 2.1 统一 Result 封装

- 所有模块对外函数统一返回 `Result<T>`。
- 成功时返回 `ErrorCode::Success`。
- 失败时返回明确错误码和错误信息。

### 2.2 推荐定义

```cpp
namespace chatroom {

template <typename T>
class Result;

using VoidResult = Result<std::monostate>;

} // namespace chatroom
```

### 2.3 约束

- 不要让函数通过“返回魔法值”表达失败。
- 不要在同一模块里混用异常返回、布尔返回和错误码返回。
- 涉及业务判断时，结果对象必须带错误信息。

---

## 3. 回调函数签名规范

### 3.1 统一别名

建议在公共头文件中统一声明回调别名：

```cpp
using ConnectCallback = std::function<void(int fd)>;
using DisconnectCallback = std::function<void(int fd)>;
using MessageCallback = std::function<void(int fd, uint16_t msgType,
											uint64_t roomId, uint64_t userId,
											const std::string& data)>;
using AuthCallback = std::function<void(int fd, uint64_t userId, uint32_t role)>;
using KickCallback = std::function<void(int fd, uint64_t operatorId, const std::string& reason)>;
using RoomStateCallback = std::function<void(uint64_t roomId, int state)>;
using ErrorCallback = std::function<void(ErrorCode code, const std::string& message)>;
```

### 3.2 约束

- 回调入参类型必须稳定。
- 不要在不同模块里对同一个事件定义不同回调形状。
- 回调参数尽量使用 `const&`，避免不必要的拷贝。
- 网络层完成 Header 解析后，再把消息类型、房间号和用户号传给业务回调。

---

## 4. 事件通知机制

### 4.1 推荐模式

- 优先使用 **观察者模式** 解耦状态变化。
- 如果事件类型较多，使用轻量 **事件总线**。
- 高频事件不要层层同步转发太多次。

### 4.2 事件范围

适合做事件通知的场景：

- 用户进入房间
- 用户离开房间
- 房间状态变化
- 弹幕到达
- 礼物到账
- 敏感词命中

### 4.3 事件约束

- 事件对象必须轻量。
- 事件总线不能反向依赖具体业务模块。
- 事件处理器必须可注册、可注销、可追踪。

---

## 5. 模块对外暴露的头文件清单

### 5.1 建议暴露的公共头文件

- `include/common/result.h`
- `include/common/error_code.h`
- `include/common/logging.h`
- `include/protocol/message.h`
- `include/protocol/message_codec.h`
- `include/net/connection.h`
- `include/net/epoll_server.h`
- `include/service/room_service.h`
- `include/service/danmaku_service.h`
- `include/service/filter_service.h`
- `include/data/repository.h`

### 5.2 暴露原则

- 对外头文件只暴露稳定接口。
- 实现细节放进 `.cpp` 或私有头文件。
- 不要让业务模块互相 include 对方内部实现。

---

## 6. 接口设计原则

- 接口要短、稳定、可组合。
- 对外接口只负责声明职责，不负责堆业务细节。
- 参数传递要尽量明确，避免“一个对象包所有内容”。
- 如果一个接口变得越来越长，说明应该拆分。
