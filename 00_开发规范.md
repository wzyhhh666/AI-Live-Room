# C++ 直播弹幕+房间系统 — 开发规范

> 本规范是项目最高优先级约束。AI 生成代码、人工编写代码、后续重构都必须优先满足本文件。

## 0. 文档使用顺序

为了让 AI 按固定流程推进开发，建议按以下顺序读取和执行文档：

1. [00_开发规范.md](00_开发规范.md)：先确认语言标准、命名、错误处理、日志和文件组织。
2. [01_项目架构设计.md](01_项目架构设计.md)：再确认整体架构、线程模型和目录结构。
3. [02_网络协议设计.md](02_网络协议设计.md)：统一 TCP 长连接、自定义协议、登录鉴权和 Protobuf 消息体。
4. [03_数据库设计.md](03_数据库设计.md)：确定 MySQL、Redis、表结构和索引。
5. [04_核心数据结构设计.md](04_核心数据结构设计.md)：再写线程池、队列、Trie、定时器等核心结构。
6. [05_接口统一标准.md](05_接口统一标准.md)：统一模块间函数风格、Result 封装和事件通知。
7. [06~09 模块设计文档](06_弹幕模块设计.md ~ 09_构建与部署.md)：进入具体模块实现前分别展开。
8. [10~12 文档](10_性能优化方案.md ~ 12_后期扩展方案.md)：基础代码稳定后再补充。

**执行原则：** 先规范、后实现；先主链路、后扩展；先数据和协议、后业务细节。

## 1. C++ 标准与编译器

- 语言标准固定为 **C++17**。
- 编译器固定为 **GCC 13+** 或 **MSVC 2022+**。
- 任何新代码必须避免使用低于 C++17 的写法，也不要擅自引入 C++20 以上语法。
- 涉及平台差异时，优先保证 Linux/GCC 可编译，其次保证 Windows/MSVC 可编译。

### 1.1 推荐编译约束

- Linux：`-std=c++17 -O2 -Wall -Wextra -Wpedantic`
- Windows：开启 `/std:c++17`，并保持警告级别较高。
- 新增警告不允许被忽略，除非有明确文档说明。

## 2. 命名规范

### 2.1 类型和变量

| 类型 | 规则 | 示例 |
|------|------|------|
| 类名 / 结构体名 / 枚举名 | PascalCase | `RoomManager`, `DanmakuMessage` |
| 函数名 | camelCase，动词开头 | `handleConnect()`, `sendBroadcast()` |
| 局部变量 | camelCase | `clientFd`, `messageCount` |
| 成员变量 | `m_` 前缀 + camelCase | `m_roomId`, `m_onlineCount` |
| 常量 | 全大写下划线 | `MAX_ROOM_COUNT`, `DEFAULT_HEARTBEAT_MS` |
| 命名空间 | 小写短横线风格不使用，统一小写单词 | `net`, `protocol`, `service`, `data` |

### 2.2 文件命名

- 头文件统一使用 `.h`。
- 实现文件统一使用 `.cpp`。
- 头文件和实现文件名称保持一一对应，例如 `room_manager.h` / `room_manager.cpp`。
- 同一模块内的文件名使用小写下划线风格，避免混合命名。

### 2.3 宏和枚举

- 宏只用于头文件守卫、编译开关、平台差异兼容，不用于业务常量。
- 业务常量优先使用 `constexpr` 或 `enum class`。
- 枚举优先使用 `enum class`，避免污染全局命名空间。

## 3. 头文件与代码组织

### 3.1 头文件守卫

- 统一使用 `#pragma once`。
- 禁止同一项目内混用 `#ifndef/#define/#endif` 和 `#pragma once`。

### 3.2 包含顺序

建议顺序如下：

1. 当前头文件对应的 `.h`
2. 标准库头文件
3. 第三方库头文件
4. 项目内部头文件

### 3.3 命名空间

- 所有业务代码必须放在项目命名空间下，例如 `chatroom`。
- 允许按模块继续分层，例如 `chatroom::net`、`chatroom::protocol`、`chatroom::service`。
- 禁止在全局命名空间中堆积业务类型和工具函数。

## 4. 智能指针使用规则

### 4.1 默认原则

- **默认优先使用 `std::unique_ptr`**，表达单一所有权。
- 只有在确实存在多方共享生命周期时，才使用 `std::shared_ptr`。
- 非所有权引用一律使用裸指针或引用，但必须清楚标注“非 owning”。

### 4.2 什么时候用 `std::unique_ptr`

适用场景：

- 连接对象由连接管理器独占持有。
- 线程池内部的任务对象。
- 协议解析器、配置对象、临时资源包装。
- 任何“一个对象只对应一个所有者”的场景。

### 4.3 什么时候用 `std::shared_ptr`

适用场景：

- 连接对象需要同时被连接管理器、房间管理器、广播任务持有。
- 延迟任务可能跨线程存活。
- 事件回调需要确保对象在回调期间不被提前销毁。

### 4.4 什么时候不用智能指针

- 纯栈对象生命周期明确时，直接用栈对象。
- 容器内仅临时使用、不会跨作用域持有时，优先值语义。
- 不要为了“看起来现代”而强行把所有对象都包成智能指针。

## 5. 错误处理策略

### 5.1 统一策略

项目运行期统一采用 **错误码 + Result 封装**，业务代码禁止把异常当作正常控制流。

### 5.2 统一返回模型

```cpp
#pragma once

#include <string>
#include <utility>

namespace chatroom {

enum class ErrorCode {
    Success = 0,
    InvalidArgument,
    ProtocolError,
    NetworkError,
    DatabaseError,
    NotFound,
    PermissionDenied,
    RateLimited,
    InternalError
};

template <typename T>
class Result {
public:
    static Result ok(T value) {
        return Result(ErrorCode::Success, std::move(value), "");
    }

    static Result fail(ErrorCode code, std::string message) {
        return Result(code, T{}, std::move(message));
    }

    bool isOk() const { return m_code == ErrorCode::Success; }
    const T& value() const { return m_value; }
    ErrorCode code() const { return m_code; }
    const std::string& message() const { return m_message; }

private:
    Result(ErrorCode code, T value, std::string message)
        : m_code(code), m_value(std::move(value)), m_message(std::move(message)) {}

    ErrorCode m_code;
    T m_value;
    std::string m_message;
};

} // namespace chatroom
```

### 5.3 异常使用边界

- 业务层、协议层、网络层返回 `Result` 或错误码。
- 只有启动期的不可恢复错误可以抛异常并在入口统一终止，例如配置文件损坏、核心资源初始化失败。
- 禁止在消息处理循环、网络收发循环中依赖异常做分支控制。

## 6. 日志规范

### 6.1 日志级别

| 级别 | 用途 |
|------|------|
| TRACE | 高频调试信息，默认关闭 |
| DEBUG | 开发调试 |
| INFO | 正常运行关键事件 |
| WARN | 可恢复异常或潜在风险 |
| ERROR | 处理失败但进程可继续 |
| FATAL | 不可恢复错误 |

### 6.2 日志格式

建议统一为：

`[时间][级别][模块][线程ID][fd/roomId][消息]`

示例：

`[2026-05-15 10:31:22.143][INFO][net][tid=12][fd=1289] client connected`

### 6.3 输出要求

- 默认输出到控制台和滚动文件。
- 生产环境必须保留按天分割或按大小分割的日志文件。
- 网络、协议、业务、数据库四类日志建议分模块输出，便于定位问题。
- 推荐使用 `spdlog` 统一封装日志接口。

## 7. 代码风格底线

- 一处功能只放在一个模块里，不在多个文件重复实现。
- 公共逻辑必须下沉到工具类或服务类，不要在 handler 中堆业务细节。
- 任何新增接口都必须先定义头文件契约，再实现 `.cpp`。
- 复杂逻辑必须先写注释，说明“为什么这么做”，而不是重复代码字面意思。
- 不允许一个文件同时承担网络、协议、业务、数据持久化四种职责。

---

## 8. 交付约束

- 所有代码应能在独立模块下编译通过。
- 任何新增依赖都必须先写进构建文档和 CMake 配置。
- 所有可配置项都必须从配置文件或启动参数读取，禁止硬编码。
- 项目内必须保持同一套命名、错误处理和日志格式。
