# C++ 直播弹幕+房间系统 — 核心数据结构设计

## 1. 设计目标

本文件约束所有高频路径中的数据结构选择。AI 写算法模块时，必须先看这里，避免结构随手乱选。

---

## 2. 线程池设计

### 2.1 线程池职责

线程池负责执行网络线程投递的业务任务，例如：

- 弹幕校验
- 敏感词过滤
- 房间状态更新
- 数据落库
- 排行榜刷新

### 2.2 任务结构体

```cpp
struct Task {
    std::function<void()> callback;
    uint64_t taskId;
    uint64_t roomId;
    uint64_t userId;
};
```

### 2.3 任务队列

- V1 统一采用 **互斥锁 + 条件变量** 的任务队列。
- 原因：实现简单、稳定、易调试。
- 后续性能优化可替换为无锁队列，但接口不变。

### 2.4 线程数量

- 默认线程数 = `CPU 核心数 * 2`。
- CPU 密集任务多时可适当增加。
- I/O 等待多时线程数可略高，但必须压测验证。

## 3. 弹幕消息队列

### 3.1 Baseline 方案

- 默认使用 **互斥锁队列**。
- 读写都必须遵循同一把锁，确保正确性。
- 适合业务初期，便于排查。

### 3.2 优化方案

- 高频广播路径可升级为 **无锁环形队列**。
- 只有在压测证明锁竞争成为瓶颈后再替换。
- 队列接口保持不变，内部实现可切换。

### 3.3 选择原则

- 项目 V1 不追求极限性能，先追求正确和稳定。
- 如果队列是广播热路径，并且压测 QPS 明显受锁影响，再上无锁方案。

## 4. Trie 树结构

### 4.1 节点定义

```cpp
struct TrieNode {
    std::array<TrieNode*, 128> children{};
    bool isEnd = false;
    int wordId = -1;
};
```

### 4.2 存储方式

- 敏感词按字符建树。
- 中文场景可先按 UTF-8 处理层进行统一封装，再进入过滤逻辑。
- 词库更新时采用重建整棵树的方式，避免原地修改带来的并发风险。

### 4.3 查询流程

1. 遍历输入文本。
2. 从每个位置向后匹配。
3. 命中敏感词则返回位置和长度。
4. 上层决定替换或拦截。

### 4.4 与 AC 自动机的关系

- Trie 树是敏感词的存储结构。
- AC 自动机基于 Trie 构建，增加 fail 指针，支持多模式串一次扫描匹配。
- 本项目最终用于线上过滤的是 AC 自动机，而不是朴素 Trie。
- 详细的 AC 自动机构建和使用方案见 [08_敏感词过滤设计.md](08_敏感词过滤设计.md)。

总结：

- 存储层：Trie，用于建树、节点定义和词库管理。
- 查询层：AC 自动机，用于基于 Trie + fail 指针的一次扫描多词匹配。

## 5. 连接管理器

### 5.1 核心职责

- 维护 fd 到 Connection 的映射。
- 提供连接查找、加入、移除、广播能力。
- 与网络层和房间层共同协作。

### 5.2 数据结构

- `std::unordered_map<int, std::shared_ptr<Connection>>` 作为主映射。
- 读多写少场景使用 `std::shared_mutex`。
- 热路径禁止频繁复制 Connection 对象。

### 5.3 关键约束

- 一个 fd 只能对应一个连接对象。
- 连接销毁必须从管理器中同步移除。
- 广播时不直接遍历原始容器修改元素。

## 6. 定时器设计

### 6.1 方案选择

本项目主方案建议使用 **时间轮**，因为心跳和超时处理属于大量、重复、固定粒度任务。

### 6.2 为什么不是只用最小堆

- 最小堆适合精确定时，但大量连接心跳场景下维护成本更高。
- 时间轮在固定间隔任务上更适合高并发连接管理。
- 连接超时、心跳踢线更符合时间轮特征。

### 6.3 使用方式

- 时间轮存放连接超时任务。
- 每次 tick 检查当前槽位。
- 连接收到心跳后刷新 slot。

### 6.4 备选说明

- 少量一次性定时任务可单独走最小堆。
- 但主业务超时检测仍以时间轮为主。

## 7. 排行榜数据结构

### 7.1 主方案

- 线上排行榜主方案使用 **Redis ZSET**。
- ZSET 适合按分数排序、实时更新和分页读取。

### 7.2 本地缓存

- 本地可以维护一个只读快照或轻量跳表缓存。
- 但最终权威数据以 Redis 为准。

### 7.3 适用场景

- 房间礼物榜。
- 用户活跃榜。
- 房间热度榜。

---

## 8. 数据结构约束

- 热路径优先选择简单、稳定、易调试的数据结构。
- 任何“更高级”的结构都必须先证明性能收益。
- 不能为了算法展示感强行引入复杂结构。
